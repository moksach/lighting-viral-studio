# Lightning Viral Studio

A local, single-purpose creator tool for turning a long phone thunderstorm video into a tight, cropped, viral-ready lightning reel.

This build is intentionally **not** a generic video editor. It solves one core job well:

1. Find the first and last visible lightning frames.
2. Remove dead dark footage.
3. Keep adjustable context before and after each lightning event.
4. Crop around the strike in a stable social-media frame.
5. Rank strikes by creator impact.
6. Export individual strike clips plus a final strongest-first reel.

## Run

You need Python 3.10+. The app uses system FFmpeg when available and falls back to the bundled `imageio-ffmpeg` binary installed from `requirements.txt`.

### macOS / Linux

```bash
cd lightning_viral_studio
./run_app.sh
```

### Windows

```bat
cd lightning_viral_studio
run_app.bat
```

Or manually:

```bash
cd lightning_viral_studio
python -m venv .venv
source .venv/bin/activate   # Windows: .venv\Scripts\Activate.ps1
pip install -r requirements.txt
streamlit run app.py
```

Optional: confirm system FFmpeg works:

```bash
ffmpeg -version
```

## Recommended first settings

For most phone thunderstorm footage:

| Setting | Value |
|---|---:|
| Lightning type | Balanced |
| Preset | Normal storm video |
| Full-FPS refinement | On |
| Keep before first light frame | 0.50 sec |
| Keep after last light frame | 0.90 sec |
| Output aspect | 9:16 vertical |
| Crop mode | Smart per-strike crop |
| Compilation order | Strongest strike first |
| Enhancement | Light contrast boost |

## When detection misses lightning

Use these adjustments in order:

1. Change preset to **Very dark sky**.
2. Lower **Bright pixel threshold**.
3. Lower **Minimum bright pixels (%)**.
4. Lower **Sudden brightness jump** and **Highlight jump sensitivity**.
5. Use **Cloud glow / behind-cloud flash** mode if the sky glows instead of showing a visible bolt.

## When false positives appear

Use these adjustments in order:

1. Change preset to **City lights / false positives**.
2. Increase **Bright pixel threshold**.
3. Increase **Minimum bright pixels (%)**.
4. Increase **Sudden brightness jump**.
5. Use **Ignore bottom of frame (%)** if cars, buildings, or streetlights appear at the bottom.
6. Keep **Suppress static lights** on.

## Export package

The ZIP generated by the app includes:

- `clips/` - individual strike clips
- `lightning_viral_reel.mp4` - final reel, if enabled
- `events_selected.csv` - exact times, crop boxes, scores
- `analysis_signal_sampled.csv` - sampled detection signal
- `analysis_signal_refined.csv` - full-FPS signal near events, if enabled
- `project_manifest.json` - versioned project metadata for future compatibility
- `caption_ideas.txt` - quick caption starters
- `README_EXPORT.txt` - export notes

## Future-safe architecture

This build is designed for surgical future changes. The active workflow is split into stable modules:

```text
app.py                   Streamlit UI only
src/lvs/contracts.py      Stable dataclasses and schema version
src/lvs/config.py         Presets, migrations, output dimensions
src/lvs/detection.py      Lightning detection and event building
src/lvs/crop.py           Stable crop calculation
src/lvs/diagnostics.py    Tuning recommendations from analysis signal
src/lvs/review.py         Preview images and editable event table helpers
src/lvs/viral.py          Scoring and reel planning
src/lvs/export.py         FFmpeg exports and project manifest
src/lvs/registry.py       Future module registry
src/lvs/video_io.py       Video probing, frame reading, synthetic test video
src/lvs/utils.py          Shared utilities
```

Rules for future modules:

1. Do not rewrite `LightningEvent` fields. Add optional fields with defaults only.
2. Add migrations in `config.py` if a setting name changes.
3. Add new modules through `registry.py` instead of hard-wiring them into detection/export internals.
4. Keep export reproducible through `project_manifest.json`.
5. Add tests or self-tests before changing detection, crop, or export contracts.

## Built-in self-test

In the sidebar, click **Use built-in synthetic test video**. It creates a short artificial lightning video and runs the exact same detection workflow. Use this when checking that a future code change did not break the lightning pipeline.
